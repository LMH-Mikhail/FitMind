# FitMind 第一版设计文档

## 信息来源与状态

- 文档状态：Draft
- 最近更新：2026-06-29
- 项目名称：FitMind
- Slogan：Dress with Intelligence
- 第一版定位：电子衣橱 + AI 衣物识别 + 规则穿搭推荐 + 用户反馈闭环
- 主要产品形态：移动端 App
- 本文档用途：
  - 作为 FitMind 第一版 MVP 的产品、交互、数据和推荐逻辑设计依据。
  - 作为后续前端、后端、AI 服务、数据库设计和测试验收的共同参考。

## 1. 项目概述

FitMind 是一个面向大众用户的移动端 AI 穿搭助手，核心目标是帮助用户管理自己的真实衣橱，并基于衣物信息、使用场景、季节天气和用户反馈生成可解释的穿搭建议。

第一版不应被设计成“训练一个复杂穿搭模型后再上线”的项目。真实 C 端 App 在冷启动阶段通常缺少足够的用户数据，因此第一版应优先跑通完整产品闭环：

```text
用户上传衣物
  ↓
AI 识别衣物标签
  ↓
用户修正标签
  ↓
形成结构化电子衣橱
  ↓
系统按场景生成穿搭
  ↓
规则引擎过滤和打分
  ↓
AI 生成推荐理由
  ↓
用户收藏 / 跳过 / 替换 / 标记已穿
  ↓
沉淀用户偏好数据
```

第一版的关键不是“模型多强”，而是建立可上线、可反馈、可积累数据的基础系统。后续第三版的通用穿搭兼容模型和个性化推荐排序，都应建立在第一版和第二版沉淀的数据之上。

## 2. 第一版核心结论

第一版穿搭推荐采用：

```text
AI 辅助识别 + 结构化标签 + 规则组合推荐 + 加权评分排序 + 用户反馈优化
```

其中：

- AI 负责识别衣物、生成标签、理解用户需求、生成推荐理由。
- 规则推荐负责组合衣服、过滤不合理搭配、计算推荐分数。
- 用户反馈负责修正偏好，为后续个性化模型准备数据。

第一版不把公开数据集训练出的穿搭推荐模型作为上线前置条件。公开数据集可以在后续用于标签体系参考、衣物识别测试和第三版模型预训练，但不应阻塞第一版上线。

## 3. 品牌定位

### 3.1 品牌关键词

```text
智能
可信赖
克制
实用
高级
个人化
```

FitMind 应该给人的感觉不是“潮流社区”或“娱乐换装工具”，而是一个理解用户衣橱、偏好和生活场景的智能穿搭系统。

### 3.2 品牌表达

- Fit：合身、适配、匹配场景、适合用户。
- Mind：理解偏好、理解风格、辅助决策。
- Dress with Intelligence：让穿衣选择更聪明，而不是更复杂。

### 3.3 应避免的方向

- 不要过度承诺“完美穿搭”。
- 不要在第一版暗示系统已经拥有成熟个性化模型。
- 不要把虚拟试衣作为第一版核心卖点。
- 不要使用容易制造身材焦虑或审美压力的文案。

## 4. 产品目标

### 4.1 第一版目标

第一版需要完成一个真实用户可用的最小闭环：

```text
注册登录
→ 上传衣物
→ AI 识别衣物信息
→ 用户确认和编辑
→ 形成电子衣橱
→ 选择穿搭场景
→ 生成 3 到 5 套穿搭建议
→ 展示推荐理由
→ 用户反馈
→ 更新偏好画像
```

### 4.2 第一版成功标准

- 用户可以顺利把自己的衣服录入系统。
- 用户可以看到结构化的电子衣橱。
- 用户可以在常见场景下生成不离谱的穿搭建议。
- 每套推荐都有清晰的理由。
- 用户可以通过收藏、跳过、替换、已穿等方式反馈。
- 后端能记录足够的数据，为后续个性化推荐准备基础。

### 4.3 第一版非目标

第一版不做：

- 真人虚拟试衣。
- 社区分享。
- 商品推荐和电商转化。
- 会员付费。
- 品牌合作后台。
- 大规模穿搭模型训练。
- 完整 AI 穿搭师对话系统。

## 5. 目标用户与使用场景

### 5.1 目标用户

| 用户类型 | 核心痛点 | FitMind 第一版价值 |
|---|---|---|
| 普通日常用户 | 每天不知道穿什么，衣服多但不会搭 | 快速生成可穿的日常搭配 |
| 通勤用户 | 上班穿搭需要得体但不想每天思考 | 根据通勤场景推荐稳妥组合 |
| 穿搭兴趣用户 | 想整理衣橱、理解自己的风格 | 管理衣物标签和收藏穿搭 |
| 衣物较多用户 | 常穿固定几件，很多衣服闲置 | 通过推荐提高衣物利用率 |

### 5.2 典型用户任务

```text
我想把衣柜里的衣服录入 App。
我希望上传图片后自动识别衣服类型、颜色和风格。
我明天上班，不知道怎么搭。
我想用这件蓝色衬衫搭一套。
这套我不喜欢，我想换一双鞋。
这套我今天真的穿了，希望系统以后更懂我。
```

### 5.3 关键使用时刻

- 早上出门前。
- 前一天晚上计划第二天穿搭。
- 周末整理衣橱。
- 准备约会、面试、旅行、聚会等场景前。

## 12. 规则推荐设计

### 12.1 规则推荐定义

规则推荐不是随机搭配，也不是完全由模型自由生成。它是把基础穿搭经验、场景要求、天气季节限制、颜色搭配原则和用户偏好写成系统规则，让系统按照规则生成和排序穿搭。

第一版用规则推荐的原因：

```text
不依赖大量训练数据
推荐结果可控
容易解释原因
冷启动可用
方便后续接入模型
```

### 12.2 支持的穿搭模板

```text
上装 + 下装 + 鞋子
上装 + 下装 + 外套 + 鞋子
连衣裙 + 鞋子
连衣裙 + 外套 + 鞋子
上装 + 半身裙 + 鞋子
上装 + 下装 + 鞋子 + 包包
```

### 12.3 第一版支持场景

第一版至少支持：

```text
通勤
休闲
约会
```

可扩展支持：

```text
运动
旅行
面试
聚会
```

### 12.4 硬规则示例

硬规则用于过滤明显不合理的搭配。

```text
运动场景不推荐高跟鞋、西装外套、长裙。
面试场景不推荐拖鞋、破洞裤、过度暴露单品。
冬季不推荐凉鞋、薄款夏季连衣裙。
夏季不推荐羽绒服、厚毛衣。
用户标记为“不推荐”的衣物必须排除。
用户指定必须使用某件衣物时，候选穿搭必须包含它。
```

### 12.5 软评分公式

第一版综合评分建议：

```text
OutfitScore =
  0.30 * 场景匹配度
+ 0.25 * 季节天气适配度
+ 0.20 * 颜色协调度
+ 0.15 * 风格一致性
+ 0.10 * 用户偏好匹配度
```

如果第一版暂时不接实时天气，`季节天气适配度` 可以先根据衣物季节标签和厚薄程度计算。实时天气联动放到第二版。

### 12.6 加分规则示例

```text
单品风格与场景匹配，加分。
颜色以基础色为主且整体不超过 3 个主色，加分。
用户曾收藏类似颜色或风格，加分。
长期未穿但仍适合当前场景的衣物，适度加分。
正式程度符合场景要求，加分。
```

### 12.7 减分规则示例

```text
大面积高饱和撞色且用户没有选择夸张风格，减分。
风格冲突明显，减分。
用户多次跳过类似类别，减分。
衣物状态为待清洗或闲置不推荐，减分或过滤。
季节厚薄不合适，减分或过滤。
```

### 12.8 推荐算法伪代码

```text
输入：
  wardrobe_items            用户衣橱中的可用衣物
  user_request              用户请求，例如通勤、约会、想穿蓝色衬衫
  user_preference_profile   用户偏好画像

输出：
  ranked_outfits            排序后的穿搭列表

步骤：
  1. 解析 user_request，得到 scene、style_goal、required_item、constraints。
  2. 读取用户可推荐状态的衣物。
  3. 按 category 和 role 对衣物分组。
  4. 根据穿搭模板生成候选组合。
  5. 执行硬规则过滤：
     - 缺少必要角色则过滤。
     - 场景禁用类别则过滤。
     - 衣物状态不可推荐则过滤。
     - 季节厚薄明显冲突则过滤。
     - 未包含 required_item 则过滤。
  6. 对每个候选组合计算分数：
     - 场景匹配度
     - 季节天气适配度
     - 颜色协调度
     - 风格一致性
     - 用户偏好匹配度
  7. 为每个得分项生成 reason_codes。
  8. 按总分从高到低排序。
  9. 去除过于相似的穿搭。
  10. 返回 Top 3 到 5 套穿搭。
  11. 调用 AI 文案服务，把 reason_codes 转成自然语言解释。
  12. 记录推荐结果和用户后续反馈。
```

## 13. 推荐理由生成

推荐结果不能只给组合，还必须说明为什么推荐。

### 13.1 内部原因码示例

```json
{
  "scene": "commute",
  "score": 91,
  "items": ["白色衬衫", "黑色直筒裤", "灰色西装外套", "黑色乐福鞋"],
  "reason_codes": [
    "scene_commute_high_match",
    "neutral_color_harmony",
    "formality_range_match",
    "straight_pants_taller_effect"
  ]
}
```

### 13.2 用户可见解释示例

```text
这套搭配适合通勤场景。白色衬衫和灰色西装外套能提升职业感，黑色直筒裤让整体线条更利落，黑色乐福鞋比运动鞋更适合上班。整体配色以黑白灰为主，稳定、干净，不容易出错。
```

### 13.3 文案要求

- 解释要具体，不要只说“很适合你”。
- 至少说明场景、颜色、风格或季节中的两个维度。
- 不使用羞辱性、评判性语言。
- 不承诺绝对显瘦、绝对显高。

| 行为 | 数据含义 |
|---|---|
| 收藏 | 用户喜欢这套穿搭 |
| 保存 | 用户未来可能会穿 |
| 标记已穿 | 用户真实采用，最强正反馈 |
| 跳过 | 用户不感兴趣，弱负反馈 |
| 替换单品 | 用户对原单品不满意，对替换单品更偏好 |
| 修改标签 | 用户修正系统理解，也是个性化信号 |

### 14.3 偏好更新规则

```text
如果用户连续收藏黑白灰穿搭：
  提高低饱和和基础色偏好权重。

如果用户多次跳过裙装：
  降低裙装推荐权重。

如果用户总是把高跟鞋替换成乐福鞋：
  在通勤或约会场景中提高乐福鞋权重，降低高跟鞋权重。

如果用户标记某套为已穿：
  提高该套中颜色、风格、类别、正式程度的偏好权重。
```

## 15. 视觉与交互方向

### 15.1 整体风格

FitMind 第一版应采用克制、清晰、实用的移动端工具风格。

```text
优先：清爽、留白、信息清晰、照片突出、操作明确
避免：过度装饰、复杂渐变、营销式首页、大面积无意义插画
```

### 15.2 色彩建议

```text
主色：深色中性或深蓝，用于品牌和关键按钮
辅助色：柔和蓝或青色，用于 AI 和智能提示
背景：浅灰白，便于突出衣物图片
警示色：仅用于错误、冲突、不可推荐状态
```

### 15.3 字体与排版

- 移动端优先，保证小屏可读性。
- 标题不宜过大，FitMind 是日常工具而不是纯营销页。
- 衣物卡片以图片为主，标签和状态为辅。

### 15.4 核心组件

```text
衣物卡片
衣物标签 Chip
分类分段控件
上传底部弹窗
AI 识别结果表单
穿搭推荐卡片
穿搭单品横向列表
推荐理由区块
反馈操作栏
替换单品选择器
偏好画像摘要卡片
```

### 15.5 状态设计

必须覆盖：

```text
上传中
识别中
识别失败
保存成功
衣橱为空
衣物类别不足
无合适推荐
推荐生成中
反馈提交中
网络错误
```

## 16. 可访问性与隐私

### 16.1 可访问性

- 图标按钮必须有可读标签。
- 关键文字不能直接压在复杂衣物照片上。
- 推荐分数和推荐理由应可被屏幕阅读器理解。
- 颜色不能作为唯一状态表达。
- 动画应克制，并考虑减少动态效果设置。

### 16.2 隐私要求

FitMind 会处理用户衣物照片和偏好数据，后续还可能处理真人照片，因此第一版就要建立隐私基础。

第一版必须提供：

```text
隐私政策入口
用户协议入口
删除衣物图片
删除账号或申请删除数据
第三方 AI 服务处理图片的说明
```

第一版应避免：

```text
默认读取整个相册
上传不必要的照片元数据
未经说明将图片用于模型训练
使用涉及身材羞辱或外貌评价的文案
```

## 17. 技术实现建议

### 17.1 移动端

可选技术：

```text
Flutter
React Native
原生 iOS / Android
```

如果追求跨平台和开发效率，建议优先考虑 Flutter 或 React Native。

### 17.2 后端

建议技术：

```text
Go
```

第一版后端建议使用 Go 承担主业务服务，包括用户、衣橱、图片上传、标签管理、规则推荐、反馈记录和 API 鉴权。Go 更适合作为长期稳定的业务后端。

AI 能力不建议全部绑定在 Go 后端内。第一版可以由 Go 后端调用外部多模态识别 API；后续如果需要自训练模型、特征提取、向量化和个性化排序，可以独立增加 Python AI 服务。

### 17.3 数据库与存储

```text
数据库：PostgreSQL
未来向量能力：pgvector
缓存：Redis
图片存储：阿里云 OSS / 腾讯云 COS / AWS S3 / MinIO
部署：Docker Compose 起步
```

数据库决策收敛为 PostgreSQL。第一版的用户、衣物、标签、穿搭、推荐记录和反馈数据都适合用关系型表建模；长期如果进入向量检索、衣物 embedding、相似搭配召回和个性化推荐阶段，可以在 PostgreSQL 上引入 pgvector，避免过早拆出独立向量数据库。

### 17.4 第一版服务拆分

第一版可以先保持简单：

```text
App 客户端
后端业务服务
AI 衣物识别服务
规则推荐模块
图片存储
数据库
```

不建议第一版就引入复杂微服务、Kubernetes、模型训练平台或推荐实验平台。

## 18. 第一版验收标准

### 18.1 用户流程验收

- 用户可以注册登录。
- 用户可以上传衣物图片。
- 用户可以看到 AI 识别结果。
- 用户可以手动修改衣物标签。
- 用户可以在衣橱中查看已添加衣物。
- 用户可以选择场景生成穿搭。
- 系统可以返回 3 到 5 套推荐。
- 推荐不足时有明确提示。
- 用户可以收藏、跳过、替换、标记已穿。

### 18.2 推荐质量验收

- 通勤场景不会推荐明显不适合通勤的搭配。
- 运动场景不会推荐高跟鞋或正式西装组合。
- 夏季上下文不会推荐明显厚重衣物。
- 每套推荐至少包含 2 条可理解理由。
- 用户指定某件衣物时，推荐应优先包含该衣物。

### 18.3 数据验收

- 衣物数据保存完整。
- AI 识别置信度被记录。
- 用户修改标签被记录。
- 每次推荐生成被记录。
- 用户反馈事件被记录。
- 用户偏好画像能根据反馈更新。

### 18.4 异常验收

- AI 识别失败时仍可手动创建衣物。
- 图片上传失败时可以重试。
- 衣橱为空时不展示空白页。
- 衣物类别不足时提示缺少哪些类型。
- 网络异常时给出可理解错误提示。
